# Internationalization scaffold (i18n / l10n)

Status: **C01 L16** — Phase-0 **multi-locale soft catalogs** (`en` + `es`) plus a
lookup helper with `SL_LOCALE` selection. This is a soft scaffold, not full
Fluent/ICU localization.

SessionLedger UI copy, CLI help, and operator docs remain mostly English inline.
The JSON catalogs establish the SSOT shape, a second-locale stub, and runtime
locale selection hooks. Fluent / ICU MessageFormat plurals stay deferred until
product demand justifies the dependency cost.

Related: [`docs/DESIGN.md`](../DESIGN.md) (product scope), ADR trail under
[`docs/adr/`](../adr/) for larger product decisions.

## Phase-0 decision

| Choice | Rationale |
|--------|-----------|
| Tiny JSON catalogs (`locales/en.json`, `locales/es.json`) | Cheap, readable, no new crates beyond existing `serde_json` |
| Rust helper (`src/i18n.rs`) | `t` / `t_locale` / `try_catalog` / `active_locale` with English fallback |
| Soft Spanish catalog | Multi-locale evidence without a Fluent runtime |
| Defer Fluent / gettext / ICU MessageFormat | Full i18n runtimes are effort **L**; catalogs unblock L16 first |
| `SL_LOCALE` env + explicit override | Operator/agent locale switch without OS locale coupling |

This is **not** a claim that the viewer or CLI already route every user-visible
string through the catalog. Inline English strings remain common; the catalog
establishes the SSOT shape and lookup API for incremental adoption.

## Catalog layout

```
locales/
  en.json          # default (embedded via include_str!)
  es.json          # soft second locale (same key set)
```

Each catalog file:

```json
{
  "locale": "en",
  "messages": {
    "viewer.app_title": "SessionLedger",
    "cli.app_about": "..."
  }
}
```

Keys use dotted namespaces (`cli.*`, `viewer.*`). Lookup:

- `session_ledger::i18n::t("viewer.app_title")` → active locale (see below)
- Missing key → English catalog → key itself
- `try_catalog("es")` → Spanish soft catalog
- `try_catalog("fr")` → `None` today

### Active locale selection

1. Explicit `t_locale(key, Some(tag))` / `active_locale(Some(tag))`
2. Else `SL_LOCALE` environment variable (primary subtag; `es-MX` → `es`)
3. Else `en` ([`DEFAULT_LOCALE`](../../src/i18n.rs))

## Future hooks (Fluent / ICU)

When richer localization becomes a goal:

1. Keep JSON catalogs as the thin SSOT **or** migrate keys to Fluent `.ftl`.
2. Optionally adopt Fluent (`fluent` / `fluent-bundle`) or ICU MessageFormat for
   plurals/selects — document as a feature flag if the dep graph grows.
3. Wire CLI (`clap`) and viewer surfaces to `t(...)` incrementally; keep
   English as the fallback chain: `requested → en → key`.
4. Consider OS locale detection only after catalogs cover the product surface.

Do **not** add gettext extract pipelines or Crowdin/Weblate wiring until at
least the soft second locale stays keyed in lockstep (SelfCheck).

## Phase-1 (Fluent catalog stub)

Phase-1 adds optional Fluent `.ftl` catalogs beside the JSON SSOT. Default
builds stay JSON-only; enable `fluent-catalog` to load embedded FTL via
`fluent-bundle`.

```
locales/
  en.json          # Phase-0 JSON SSOT (always embedded)
  es.json          # Phase-0 soft second locale
  en.ftl           # Phase-1 Fluent stub (hyphenated message ids)
  es.ftl           # Phase-1 Fluent stub (same keys as en.json)
```

JSON dotted keys map to Fluent message ids by replacing `.` with `-`
(`viewer.app_title` → `viewer-app_title`). Lookup:

- `session_ledger::i18n_fluent::t_fluent("viewer.app_title", Some("es"))`
- Feature **off** → delegates to [`i18n::t_locale`](../../src/i18n.rs) (JSON)
- Feature **on** → Fluent bundle first, then JSON fallback chain

Enable in `Cargo.toml`:

```toml
[features]
fluent-catalog = ["dep:fluent-bundle", "dep:unic-langid"]
```

Full viewer / CLI string migration to Fluent remains deferred; Phase-1 only
proves catalog shape, optional runtime, and SelfCheck parity with JSON keys.

## Machine verification (SelfCheck)

Hermetic catalog + helper smoke (no daemon, no network, no Fluent runtime):

```powershell
pwsh ./scripts/i18n-check.ps1 -SelfCheck
```

`-SelfCheck` asserts `locales/en.json` + `locales/es.json` share required keys,
`src/i18n.rs` exposes `t` / `t_locale` / `try_catalog` / `SL_LOCALE` /
`DEFAULT_LOCALE`, this doc keeps the Phase-0 decision and Fluent/ICU future
hooks, and the Rust wrapper test is present.

Fluent catalog stub (Phase-1; no cargo required):

```powershell
pwsh ./scripts/fluent-i18n-check.ps1 -SelfCheck
```

`-SelfCheck` asserts `locales/en.ftl` + `locales/es.ftl` exist and share JSON
key parity, `src/i18n_fluent.rs` exposes `t_fluent`, optional `fluent-catalog`
feature is documented, and the Rust wrapper test is present.
